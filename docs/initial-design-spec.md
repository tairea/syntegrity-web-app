vue 3 project using supabase as backend

ui views (single file components):

1 - Initiate: start syntegrity session button and join button. start -> Driving question modal "Please enter the driving question for this syntegrity session" as well as invite link to session and info icon with good DQ examples tooltip. join -> session id# input that takes user to already created session.

2 - User Profile: ability to add image and name and discord handle.

3 - Lobby: 3d visualisation of the syntegrity geometric shape (using three.js) that best suits the number of people currently in the lobby (see: other-geometries-for-varying-group-sizes.md). shape rotating. graph vertices should be textboxes with topic placeholder eg. "topic n". graph edges should have a circular node placed on the middle of the edge with the person profile picture filling the circle, and name as a label just below it. as more users enter the lobby the geometric shape should update. on the left and right had sides of the screens (desktop) are cards of each person in the lobby, so you can see quickly who is in the lobby. (on mobile) just show graph with a participant number total as a chip on top right corner, when clicked a modal appears to scroll through all people in the lobby. top center is the Driving Question. button at the bottom center say "Enter Problem Jostle". Users can progress from Lobby to Problem Jostle any time. Users can click their profile cards to edit their profile.

4 - Problem jostle state: same as 3, but different state. in this state side member cards disappear, the geometric shape morphs into one circle ring with people nodes evenly distributed around the ring (ring should extend out to full edges of the viewport, to create maximum space on the inside of the ring). in the middle of the ring is the Driving Question. at the bottom center of the screen is a text input where paticipants can enter their "Statements", give some guidance here about statement generation. entered statements appear inside the ring as a floating statement chip (with physics) in realtime to all users. hovering/clicking over a chip draws a dashed dim line to the person who created it. hovering/clicking over a person draws dashed dimmed lines to all the chips they created. chip creators can delete their chips (trash icon on hover). every 5 seconds an llm api call is made with all statements to identify topic clustering (as they relate to the Driving Questions) (ui: text in bottom right order, cybernetic clustering in 5,4,3,2,1), chips move according to this clustering, a label is created in the middle of the chip cluster naming the cluster. 5 second clustering llm api call only triggers when new statements are added, or edited. clustering updates should update the cluster name to best reflect its chips. clustering calls dont need to keep happening if no new statements are entered. each cluster has a chat icon (with a notification badge showing the number of messages in the chat the user hasnt seen), that when clicked opens a chat sidepanel on the left (shrinking the graph width) with the graph now solely focusing on the contents of that cluster (everything else removed). participants can chat in the side panel. and still generate statements (for that cluster only) using the bottom center text input (relative to graph width). llm call for cluster naming every 5 seconds triggered only on new chips added to the cluster. clicking an X close button at the top of right corner of the chat side panel returns the user back to the main problem jostle view. hovering over your person node reveals a "Ready to Vote" button that when clicked puts a green border around your circle with the status label below in green that says "Ready". Once all participants are ready a modal appears saying "Moving to Voting in 5,4,3,2,1". llm clustering should aim to surface at minimum the number of syntegrity geometric shape vertices, even if only one chip or two chips in the cluster. More topic clusters are ok, but the system needs topic clusters for the number of vertices at least. 

5 - Voting: this ui should have 20vh section for the top voted topics, there should (N - number of topics aka vertices of the finalised syntegrity geometric shape) empty card slots with their respective number above the slot. Below the top voted cards section is the 80vh voting topics space, where all the clustered topic names are now cards (with llm generated rationale summarising participant chip entries in the previous step) to be voted on. there should be a left side panel that lists all participants (active user at the top) and next to the image and name are 5 dots that represent their 5 votes. As they click cards in the voting space, their mini profile image gets added to that clicked card representing one of their votes and a dot is removed from their name in the leftpanel list. once they have clicked 5 cards, their name is the listpanel is greyout out showing they have finished voting. Clicking a card, adds your image-vote at the bottom of the card but also adds an X (visible to that user only), they click this X to remove their vote from that card. Clicking a card more than once adds another vote, and another X. The topic cards with the most votes fill up the top voted slots according to the total amount of votes they have. Once all participants votes are cast a modal appears saying "Moving to Topic Preference in 5,4,3,2,1". Merging topics feature: any user can drag a topic card and release on top of another topic card to trigger a merge topic request. This pops out a merge topic request card from the right hand edge of the screen that states "{{name}} proposes to merge {{topic A}} with {{topic B}}" with a thumbs up green icon and a thumbs down green icon in each bottom corner of the request card. clicking either icon shows a count. center to the icons is a 30sec timer countdown, the decision with the most counts after the timer expires get executed eg. if approved, a new topic card is created (with a new llm generated title and combined rantionale) with the previous combined cards removed. if rejected topic card stays the same. anyone can click the 30sec coundown timer to add another 30 seconds. merge topic request card disappears once decision is made.

6 - Topic preference: ui is N-topic-number of the top voted cards from the previous step stacked vertically. the user can drag-and-drop the cards to order them according to preference (top being most prefferred, bottom being least preffered). have a description somewhere that explains what to do. have the same leftside panel (as in step 5) with a "Save Preferences" button next to each persons name, which clicked a state of "Ready to proceed" appears in place of the button. Allow the user to click back to undo their ready state. Once all participants preferences are saved, a modal appeasr saying "Moving to Syntegrity Graph in 5,4,3,2,1".

7 - Syntegrity graph: this view shows the completed syntegrity graph with the vertices topic textboxs populated with the topic names, and the connected edges with particpants circle node image and name on the middle of the edge. each participant is connected to two topics as a participant that they had some preference for, and they are also connected to two topics as a critic. Some sort of algorithm or llm processing has to be done to try to match partcipants to their preferred topics as best as possible. please explore both methods, creating them as separate files for easier iteration (eg. /util/7-role-assignment-algorithm.ts & /util/7-role-assignment-llm.ts). As we are in prototyping stage, please execute both methods with the step 6 data, and have a toggle button in the ui to switch between their respective output syntegrity graphs (this is for our current prototype testing phase only). Features of the syntegrity graph view are: 1) focus graph view: the ability to click on a person or topic to bring them to the center. eg: if a person is clicked, the graph rotates to bring them to the center hand have their connected vertices/topics horiztonally to the left and right of them, every other part of the graph is dimmed. if a topic is clicked, the graph rotates to bring that topic to the center, with its connected people visible around the edges, topic and connected participants visible, all other graph parts dimmed. the person or topic focused item also has a "show critque {{rotate icon}}" which shows the critique (flipped mode) for the focused person or topic. In critque mode when a person is focused the connected topics will temporarily switch to the person's critiquing topics. In critque mode when a topic is focused the connected people will temporarily switch to the topic's critiquing people. In critique mode edges should be coloured red so to easily show the user is in critque mode, clicking the critque button switches it back to normal. 2) scheduler. there should be a utility script (eg. /util/scheduler.ts) that works out the team discussions. ui the schedule should populate a leftside panel that can be toggle to expand and hide, and should show the users personal schedule as will as the full schedule of all team discussions of the syntegrity session.
